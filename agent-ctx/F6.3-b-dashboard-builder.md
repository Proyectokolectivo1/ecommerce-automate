## Implementation summary
- `'use client'` directive.
- Types: `AlertRow`, `AlertListResponse`, `AlertStats`, `EvaluateResponse`, `ResolveResponse`.
- Constants:
  - `SEVERITY_META` — icon (AlertOctagon/AlertTriangle/Info) + color classes (rose/amber/zinc) + label.
  - `ALERT_META` — exactly as specified (5 types, label, lucide icon, badge class light/dark).
  - `TYPE_OPTIONS`, `SEVERITY_OPTIONS`, `STATUS_OPTIONS` for selects.
  - `PAGE_SIZE = 25`, `AUTO_REFRESH_MS = 10_000`, `STATS_STALE_MS = 30_000`.
- Subcomponents: `SeverityIcon`, `AlertTypeBadge`, `AlertCard`, `AlertsListSkeleton`, `FilterSelect` (small reusable wrapper).
- Main page:
  - `useSession` → role. `isAdmin` hides/shows "Evaluar ahora"; `canResolve` (ADMIN/GERENCIA/SERVICIO) hides/shows "Resolver".
  - Session loading → full-page skeleton.
  - `useQuery(['alerts-stats'])` with `staleTime 30s` and `refetchInterval` based on `autoRefresh`.
  - `useQuery(listQueryKey)` with `placeholderData: prev`, `refetchInterval` when `autoRefresh`.
  - `useMutation` for resolve → `POST /api/alerts/[id]/resolve`. On success: toast "Alerta resuelta" + invalidate `['alerts']` and `['alerts-stats']`. Per-row `isResolving` flag (matches `variables === alert.id`).
  - `useMutation` for evaluate → `POST /api/alerts?evaluate=true` (ADMIN). On success: toast with `evaluated/created/duplicates`, info or success variant. On error: red toast.
  - `useRealtime({ onAlert })` → invalidates both queries + toast (toast.error for CRITICAL, toast.info otherwise).
  - Filters: search input (debounce 300ms), type Select, severity Select, status Select (ACTIVE→`resolved=false`, RESOLVED→`resolved=true`), "Limpiar" button when any filter active, Switch "Auto-actualizar (10s)".
  - Header: title + subtitle + count badge + admin-only "Evaluar ahora" button OR read-only amber note for non-ADMIN.
  - KPI row (4 KPICard): Total/Bell, Activas/AlertCircle (amber border hint when >0), Críticas/AlertOctagon (rose border hint, subtitle "Requieren atención"), Resueltas/CheckCircle2 (emerald border hint).
  - Error banner when stats or list query fails.
  - List: vertical cards (not table). Each card: severity icon left, type badge + entity + relative time, message (medium, line-through+opacity-60 when resolved), "Resuelta {relative}" emerald row when resolved, "Resolver" button footer when active + canResolve.
  - Empty state: "No hay alertas registradas" + Inbox icon + contextual help + "Limpiar filtros" button when filters active.
  - Scroll container: `max-h-[calc(100vh-26rem)] overflow-y-auto [scrollbar-width:thin]` for long lists.
  - Footer note when list has more items than `PAGE_SIZE`.

## Notes / decisions
- Colors used: rose/amber/teal/violet/zinc/emerald — no indigo/blue. ✅
- The "Evaluar ahora" button is ADMIN-only per spec; non-ADMIN sees a read-only amber pill (consistent with `impresion/page.tsx` pattern).
- The "Resolver" button is hidden for BODEGA (API rejects with 403). Hidden vs. error toast → hidden is friendlier UX.
- Auto-refresh Switch uses `refetchInterval` on both queries (10s) as specified.
- `useRealtime` `onAlert` triggers `invalidateQueries` for both `['alerts']` and `['alerts-stats']` plus a contextual toast.
- `placeholderData: (prev) => prev` keeps the list stable during refetches (no flicker).
- Per-row resolve loading: compares `mutation.variables === alert.id` so only the clicked button spins.
