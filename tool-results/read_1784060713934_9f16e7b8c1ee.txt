    17→# 0\. INSTRUCCIONES IMPERATIVAS PARA LA IA DESARROLLADORA
    18→
    19→Este documento es la fuente principal de verdad del proyecto. La IA que diseñe o programe la solución deberá cumplir estas reglas:
    20→
    21→1. No inventar endpoints, firmas, payloads, códigos de estado, capacidades, precios ni comportamientos de Shopify, Wompi, Mastershop, Meta, Google Ads, TikTok Ads o cualquier tercero.
    22→2. Cuando una integración no tenga documentación suficiente, crear primero:
    23→
    24→   * interfaz de proveedor;
    25→   * adaptador;
    26→   * mock;
    27→   * fixtures;
    28→   * pruebas de contrato;
    29→   * feature flag;
    30→   * modo simulación;
    31→   * kill switch.
    32→3. No comenzar escribiendo toda la plataforma. Trabajar por verticales funcionales pequeñas, demostrables y reversibles.
    33→4. No declarar una tarea como terminada si:
    34→
    35→   * no compila;
    36→   * falla lint;
    37→   * falla typecheck;
    38→   * fallan pruebas;
    39→   * no cumple criterios de aceptación;
    40→   * no se revisó idempotencia;
    41→   * no se revisaron errores y observabilidad;
    42→   * no se actualizó documentación.
    43→5. Toda decisión no definida deberá registrarse como:
    44→
    45→   * `SUPUESTO`;
    46→   * `DECISIÓN PENDIENTE`;
    47→   * `RIESGO`;
    48→   * `BLOQUEANTE`.
    49→6. Nunca colocar credenciales, tokens, teléfonos, correos operativos o secretos directamente en el código.
    50→7. Toda operación externa debe ser trazable mediante:
    51→
    52→   * `correlationId`;
    53→   * `causationId`;
    54→   * `orderId`;
    55→   * `storeId`;
    56→   * `provider`;
    57→   * número de intento;
    58→   * fecha;
    59→   * resultado.
    60→8. Toda operación susceptible de repetición debe ser idempotente.
    61→9. Toda integración debe tolerar:
    62→
    63→   * reintentos;
    64→   * webhooks duplicados;
    65→   * webhooks tardíos;
    66→   * eventos fuera de orden;
    67→   * timeout;
    68→   * respuesta perdida;
    69→   * caída temporal del proveedor.
    70→10. Las acciones destructivas o irreversibles requieren autorización humana explícita:
    71→
    72→    * migraciones destructivas;
    73→    * eliminación de datos;
    74→    * cancelación masiva;
    75→    * cambios de permisos;
    76→    * despliegue a producción;
    77→    * rotación de secretos;
    78→    * cambios en reglas financieras;
    79→    * activación de automatizaciones reales en Mastershop.
    80→11. La IA debe revisar documentación oficial vigente antes de implementar cada integración.
    81→12. Si la documentación real contradice este archivo, no alterar silenciosamente el alcance: registrar la incompatibilidad y proponer una decisión.

    99→# 1\. VISIÓN DEL PRODUCTO
   100→
   101→Construir una plataforma empresarial que centralice y automatice el ciclo operativo de ecommerce:
   102→
   103→```text
   104→Shopify
   105→  → recepción del pedido
   106→  → validación y normalización
   107→  → clasificación del método de pago
   108→  → cobro anticipado del transporte cuando sea contraentrega
   109→  → confirmación de Wompi
   110→  → creación del despacho en Mastershop
   111→  → selección de transportadora
   112→  → generación de guía
   113→  → actualización de Shopify
   114→  → impresión automática local
   115→  → seguimiento logístico
   116→  → entrega o devolución
   117→  → métricas, rentabilidad y auditoría
   118→```
   119→
   120→La plataforma debe eliminar la validación manual previa a la creación de guías. Los errores no deberán perder pedidos ni requerir búsquedas manuales en múltiples sistemas.

   124→# 2\. OBJETIVOS
   125→
   126→## 2.1 Objetivos funcionales
   127→
   128→* Recibir pedidos de una o múltiples tiendas Shopify.
   129→* Identificar pago anticipado y pago contraentrega.
   130→* Cobrar anticipadamente el transporte de todos los pedidos contraentrega.
   131→* Enviar automáticamente el enlace mediante Meta WhatsApp Cloud API.
   132→* Procesar la confirmación de Wompi.
   133→* Crear automáticamente el despacho en Mastershop.
   134→* Generar, almacenar y enviar la guía a impresión.
   135→* Actualizar Shopify con etiquetas, notas, fulfillment y tracking.
   136→* Monitorear estados de Mastershop.
   137→* Administrar conversaciones de WhatsApp.
   138→* Ofrecer dashboard operativo, financiero y publicitario.
   139→* Mantener histórico completo.
   140→* Permitir conciliación y reproceso seguro.
   141→* Gestionar integraciones, usuarios, permisos, plantillas, tarifas e impresoras.
   142→
   143→## 2.2 Objetivos no funcionales
   144→
   145→* Soportar al menos 500 pedidos por día.
   146→* Procesar cada pedido exactamente una vez a nivel de efecto de negocio, aunque la infraestructura use entrega “al menos una vez”.
   147→* Recuperarse de errores temporales.
   148→* Tener tiempo objetivo de recuperación operativa máximo de 35 minutos.
   149→* Mantener backups nocturnos.
   150→* Conservar información histórica y auditoría por hasta 10 años según la política definida.
   151→* Operar inicialmente con presupuesto reducido.
   152→* Ser desplegable mediante Docker Compose.
   153→* Mantener seguridad, auditabilidad, extensibilidad y capacidad de observación.
   154→
   155→## 2.3 Fuera del alcance inicial
   156→
   157→* SaaS multiempresa.
   158→* Aplicación móvil nativa.
   159→* Kubernetes.
   160→* Microservicios independientes.
   161→* Administración de inventario como fuente de verdad.
   162→* Checkout propio de tarjetas.
   163→* ZPL nativo de Mastershop.
   164→* Automatización de reembolsos.
   165→* IA predictiva en el MVP.
   166→* Staging dedicado, salvo que el presupuesto cambie.
   167→* Correo como canal de pago en la primera versión.

   569→## 8.1 Pedido pagado anticipadamente
   570→
   571→```mermaid
   572→sequenceDiagram
   573→    autonumber
   574→    participant S as Shopify
   575→    participant API as API Webhook
   576→    participant DB as PostgreSQL
   577→    participant Q as BullMQ
   578→    participant M as Mastershop
   579→    participant OS as Object Storage
   580→    participant P as Print Agent
   581→
   582→    S->>API: orders/create o evento relevante
   583→    API->>API: validar HMAC, timestamp e idempotencia
   584→    API->>DB: guardar webhook y pedido mínimo
   585→    API->>Q: enqueue sync-order
   586→    API-->>S: 200 rápido
   587→

   791→
   792→## 9.3 Reglas
   793→
   794→* Ningún estado se cambia directamente desde el frontend.
   795→* El frontend invoca casos de uso autorizados.
   796→* Cada transición:
   797→
   798→  * valida estado origen;
   799→  * valida permiso;
   800→  * registra actor;
   801→  * registra causa;
   802→  * emite evento;
   803→  * actualiza `order\_state\_history`.
   804→* `ENTREGADO` define venta reconocida.
   805→* `EN\_TRANSITO` define pedido enviado.
   806→* `DEVUELTO` define devolución.
   807→* Los eventos tardíos no pueden retroceder el estado sin una regla explícita.
   808→* Los mapeos de Mastershop son configurables.

  1501→# 12\. CONTRATOS DE WEBHOOK
  1502→
  1503→## 12.1 Reglas comunes
  1504→
  1505→Todo endpoint de webhook debe:
  1506→
  1507→1. leer body crudo cuando la firma lo requiera;
  1508→2. validar firma;
  1509→3. validar timestamp;
  1510→4. generar `correlationId`;
  1511→5. registrar payload redacted;
  1512→6. resolver idempotencia;
  1513→7. responder rápido;
  1514→8. procesar en cola;
  1515→9. no ejecutar lógica prolongada dentro de la petición;
  1516→10. devolver código apropiado.
  1517→
  1518→## 12.2 Shopify
